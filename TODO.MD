## Fase 47: Hierarquia de Roles вҖ” Dev (supremo) vs Admin (restrito) вң… (v2.6.17)

### Contexto
Atualmente a role `admin` tem acesso total via `get_admin_client()` (service_role, bypass RLS): deleta dados de qualquer usuГЎrio, gerencia contas, promove outros admins. Isso concentra **poder excessivo** em uma Гәnica role вҖ” se um admin for comprometido, todo o sistema Г© comprometido.

A soluГ§ГЈo Г© quebrar em 3 roles com responsabilidades distintas:

| Role | Acesso a dados prГіprios | Acesso a dados alheios | Gerenciar usuГЎrios | Export total | Deletar usuГЎrios |
|------|------------------------|----------------------|-------------------|-------------|------------------|
| `dev` | вң… | вң… (supremo) | вң… | вң… | вң… (qualquer um) |
| `admin` | вң… | вқҢ (sГі dele) | вң… (nunca p/ dev) | вң… | вқҢ (sГі a prГіpria conta) |
| `usuario` | вң… | вқҢ | вқҢ | вқҢ | вқҢ (sГі a prГіpria conta) |

**PrincГӯpio:** `admin` gerencia usuГЎrios (promove/rebaixa entre admin/usuario) e exporta dados totais, mas no CRUD de dados (cilindro, elemento, leitura, pressao, amostra) sГі enxerga os prГіprios вҖ” como `usuario`. `dev` substitui o poder absoluto que `admin` tinha antes, incluindo deletar qualquer dado ou conta.
